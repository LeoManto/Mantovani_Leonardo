Introduction

Our motto:
there is no code without a project, no project without problem analysis and no problem without requirements.

Requirements

Design and build a software system that makes a robot is able to walk along the boundary of a rectangular, empty room.

Requirement analysis

After interviewing client, meanings he associates with nouns have been clarified:

  • room: a conventional room, as found in all buildings
  • boundary: room's perimeter, physically delimited by solid wall
  • robot: a device able to moving after receiving commands by network, as reported in VirtualRobot2021.html

Regarding to actions (verbs):

  • walk: the robot have to move forward, near room's walls

Robot is a system composed by three macro compnent:

  • EDGE component
  • FOG component
  • CLOUD component

    • Robot will be remotely controlled, then system will be distributed and commands will send to robot through a wifi network

    A first user story

    Like user, I place the robot in the HOME cell (facing sud), and then activate a system that send to robot (using a WiFi network) movements commands.
    User can't stop activity execution: system must terminate independently after the completion of activity

    At the end of system's execution, I expect that robot performed (only one time) path shown in figure.

    Verification of expected results (Test plans)

    It is necessary to verify that path performerd by robot is as expected.
    Verification of correctness of path, it must be carried out by software, without an human user's interaction

    Problem analysis

    Aspects noted

    1. It's about realizing adistributed system composed by two macro-component:
      • robot (virtual) provided by committent
      • Our application (boundaryWalk) that send correct commands to robot so as to satisfy requirements
    2. The robot can be moved by network in two diffent ways, like described in VirtualRobot2021.html: commands:
      • sending messages to port 8090 with HTTP POST protocol
      • sending messages to port 8091 using websocket technology
    3. As there are a lot of library in a moltitude of programming languages that permit sending of these command, it's not identified any significative abstraction-gap on operation plan.
      However exist anabstraction-gap on conceptual plan: there are two possible technologies to communicate with robot, and both require appropriate supports. Anyhow conceptually it's about setting up a request-response interaction schema.
    4. Estimate that a first application prototype, should be realized in one working week (at most).

    Individuated problems

    1. Abstraction of interaction

      Software system should be made more possible indipendent from communication protocol used for interaction with WEnv Il sistema software dovrebbe essere reso il pi√π possibile indipendente dal protcollo di comunicazione usato per la interazione con WEnv.
      Projectist could use some design pattern to make application level as much as possible invariant compared to communication protocol, that could be selected in a system's configuration stage.

    2. Testing, but not only

      Goal of difine an automatizable TestPlan introduce one new requirement : make apllication observable, or rather make it verifiable effects, by one machine.
      In particular, it's not enough to induce robot to move; whithout camer and/or appropriate environmental sensors, one way to proceed is to infer knowledge by commands sended and by the responses obtained from WEnv and represent explicitly this knowledge.

      we would say that we need to create inside application, a kind of 'mind map' able to provide a model of robot's actions, upon which a testing application can "think" to make verifications.

      In particular, given the informations about robot usage, we can prefigure that problem's solution constit of this algorithm: 
      The robot start from position HOME, facing sud (DOWN) 
per 4 volte: 
	I send commands moveForward to robot until it hit wall in front of him 
	and then i send a rotation command turnLeft
